# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет**: `S12-hw-dataset.csv` (почасовые значения `target`).
- **Temporal split**: train (70%), validation (15%), test (15%) в строго хронологическом порядке.
- **Сравнивались подходы**:  
  - `B1` (naive‑last)  
  - `B2` (moving average с окном 7)  
  - `B3` (Ridge регрессия на лаговых, rolling и календарных признаках)  
  - `R1` (GRU с окном 24)
- **Лучший подход**: `R1 (GRU)` по валидационной MAE.

## 2. Среда и воспроизводимость

- Python: 3.10
- pandas / numpy / scikit‑learn: последние стабильные
- torch: 2.0.1
- Устройство: CPU (или GPU при наличии)
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи

- **Датасет**: `S12-hw-dataset.csv`
- **Целевая колонка**: `target`
- **Частота**: hourly
- **Горизонт прогноза**: 1 шаг (следующий час)
- **Размер датасета**: 4319 строк (в полной версии)
- **Диапазон дат**: 2025-01-01 00:00:00 – 2025-06-29 23:00:00
- **Дополнительные признаки**: отсутствуют
- **Комментарий**: Ряд имеет ярко выраженную суточную сезонность (пик днём, спад ночью) и более слабую недельную (выходные ниже). Наблюдаются резкие выбросы. Пропусков нет. Прогноз осложнён высокой волатильностью.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Train**: 2025-01-01 00:00 – 2025-04-12 15:00 (≈70%)
- **Validation**: 2025-04-12 16:00 – 2025-05-08 15:00 (15%)
- **Test**: 2025-05-08 16:00 – 2025-06-29 23:00 (15%)

**Почему random split некорректен?**  
Во временных рядах порядок наблюдений имеет значение. Случайное перемешивание приводит к тому, что будущие данные могут попасть в обучающую выборку, а прошлые – в тестовую. Это даёт нереалистично оптимистичную оценку качества и не проверяет способность модели прогнозировать будущее.

### 4.2. Признаки для baseline-моделей

- **Лаги**: 1, 7, 14
- **Rolling**: среднее и стандартное отклонение за 7 часов
- **Календарные**: час, день недели, месяц
- **Обработка пропусков**: строки с NaN (из‑за лагов) удалены.
- **Масштабирование**: `StandardScaler`, обучен только на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (naive‑last)**: прогноз = последнее значение train (константа).
- **B2 (moving‑average)**: прогноз = среднее за последние 7 часов train.
- **B3 (ridge‑lag‑features)**: `Ridge(alpha=1)` на 8 признаках.
- **R1 (gru‑forecast)**: GRU (2 слоя, hidden=64, dropout=0.2) на последовательности `target` длины 24.

**Основная метрика выбора**: MAE.  
**Дополнительные метрики**: RMSE, MAPE.  
**Параметры R1**: `window_size=24`, `batch_size=64`, эпохи до 50 (early stopping), `lr=0.001`, Adam.  
Лучшая модель сохранена по минимальной `val_MAE`.

## 7. Анализ

Наивный baseline ожидаемо слаб на волатильном ряде. Moving average сглаживает, но не ловит сезонность. Ridge на лагах и rolling-статистиках смог уловить суточную периодичность – значительное улучшение. GRU, благодаря способности моделировать долгосрочные зависимости, немного превзошёл Ridge, особенно на резких перепадах.

**Возможные утечки данных** в такой задаче: использование будущих значений при создании признаков (например, rolling-среднее, рассчитанное с заглядыванием вперёд) или случайное перемешивание при split. Мы их исключили: все признаки построены только по прошлым данным, split – хронологический.

**Почему выбор по validation, а не по test, принципиален?**  
Test должен использоваться только один раз – для финальной оценки выбранной модели. Если бы мы настраивали гиперпараметры по test, то получили бы переобучение под конкретный тестовый отрезок и не смогли бы честно оценить обобщающую способность.

Наиболее показательные ошибки модели – в моменты резких падений (выходные/праздники), где даже GRU даёт сбой. Однако в целом GRU показал более стабильный прогноз на пиках.

## 8. Итоговый вывод

Для данной задачи GRU дал небольшое, но ощутимое преимущество перед линейной моделью на признаках (снижение MAE на ~1.7). Это объясняется наличием сложных нелинейных паттернов (например, форма дневного пика меняется ото дня к дню). Переход к нейросетям имеет смысл при наличии достаточной истории (у нас >3000 точек). Главный урок: **корректная валидация по времени – обязательное условие для временных рядов**, иначе результаты будут оптимистично смещены.
